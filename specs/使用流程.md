# SpecForge 工作流

## 项目启动流程
```
/project-requirements-clarification [想法描述]
/project-product-overview
/project-tech-stack
/framework-design [架构范围]              ← 🆕 架构设计（可选，复杂项目推荐）
/project-structure
/project-dev-standards
/project-roadmap-planning
/project-initialization
```

**💡 何时需要 /framework-design？**

✅ **推荐使用**：新项目启动、重大架构调整、技术选型变更、系统重构  
❌ **可跳过**：简单项目、小型工具、原型验证

**📌 /framework-design 说明**

**核心价值**：将模糊的想法或明确的约束转化为清晰的、可执行的**架构设计方案**

| 维度 | 内容 |
|------|------|
| **触发场景** | 新项目启动、架构调整、技术选型变更、系统重构、模块划分 |
| **输入要求** | 产品概述、技术栈约束、性能要求、扩展性需求、部署环境 |
| **输出产物** | 架构设计方案 (`specs/架构设计.md`)、模块划分图、数据流图、技术选型清单 |

**思考维度**：
1. **架构风格选择**：单体 vs 微服务 vs 分层架构 vs 事件驱动
2. **模块职责边界**：高内聚低耦合的模块划分
3. **数据架构设计**：数据模型、存储方案、缓存策略
4. **技术组件选型**：框架、数据库、中间件的选择与理由
5. **关键质量属性**：性能、可靠性、安全性、可扩展性保证

**使用示例**：
```bash
/framework-design LocalVideoServer 整体架构    # 新项目
/framework-design 视频处理子系统                 # 子系统设计
/framework-design 微服务迁移可行性               # 架构评估
```

## 功能开发流程
```
/feature-requirements-clarification [功能描述]
/interface-design [接口类型或功能区域]          ← 🆕 接口设计（可选，涉及 API 时推荐）
/feature-tech-design
/feature-task-planning
/feature-implementation 完成[功能名]的第[N]阶段
```

**💡 何时需要 /interface-design？**

✅ **推荐使用**：定义新 API 端点、多模块协调接口、外部系统集成、详细 API 文档  
❌ **可跳过**：简单内部函数、遵循现有模式的小修改

**📌 /interface-design 说明**

**核心价值**：将功能需求转化为清晰、一致、可实现的**接口规范契约**

| 维度 | 内容 |
|------|------|
| **触发场景** | RESTful API 设计、模块间接口定义、数据格式约定、错误码体系、API 文档编写 |
| **输入要求** | 功能需求文档、业务规则、调用方需求、现有接口风格参考、安全性能要求 |
| **输出产物** | API 文档 (`specs/API文档.md`)、接口契约规范、错误码体系、数据模型定义、调用示例代码 |

**思考维度**：
1. **接口识别与分类**：CRUD 操作、查询操作、业务操作、辅助操作
2. **RESTful 规范设计**：HTTP 方法、URL 路径、状态码、分页排序
3. **请求/响应格式**：JSON 结构、字段命名、嵌套对象、版本兼容
4. **数据模型与校验**：字段定义、类型约束、必填/可选、取值范围
5. **错误处理机制**：业务错误码、异常场景、重试策略

**接口规范模板示例**：
```markdown
### 获取视频列表

**端点**: `GET /api/videos`
**描述**: 获取所有视频，支持搜索和分类筛选

**Query Parameters:**
| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| search | string | 否 | - | 搜索关键词 |
| category | string | 否 | - | 分类名称 |

**成功响应 (200):**
```json
[
  {
    "id": 1,
    "title": "video.mp4",
    "path": "/path/to/video.mp4",
    "category": "movies",
    "size": 1024000000
  }
]
```
```

**使用示例**：
```bash
/interface-design 视频管理系统 API              # 完整 RESTful API
/interface-design 黑名单管理 API 文档            # 补充接口文档
/interface-design 视频扫描器回调接口             # 内部模块接口
/interface-design 全局错误码标准                 # 错误码体系
```

## 功能变更流程
```
/feature-evolution [功能名] [变更描述]
/interface-design [受影响的接口]                ← 🆕 如需更新接口规范
/feature-implementation 完成[功能名]的变更任务
```

**💡 变更流程中的 /interface-design**

当功能变更影响现有接口时，建议先更新接口规范：
- 新增 API 端点 → 补充接口文档
- 修改请求/响应格式 → 更新接口契约
- 废弃旧接口 → 标记版本兼容性

## 💡 最佳实践提示

### 流程选择指南

#### 场景A：全新项目启动
```
/project-requirements-clarification
    ↓
/project-product-overview
    ↓
/project-tech-stack
    ↓
/framework-design ✅ 推荐（复杂项目）
    ↓
/interface-design ✅ 推荐（如果有核心 API）
    ↓
/project-structure
    ↓
/project-dev-standards
    ↓
/project-roadmap-planning
    ↓
/project-initialization
```

#### 场景B：现有项目新增功能
```
/feature-requirements-clarification
    ↓
/interface-design ⚠️ 可选（如果涉及新 API）
    ↓
/feature-tech-design
    ↓
/feature-task-planning
    ↓
/feature-implementation
```

#### 场景C：架构重构
```
/framework-design ✅ 必须（新架构方案）
    ↓
/interface-design ✅ 推荐（更新接口以适应新架构）
    ↓
/feature-evolution（渐进式迁移）
    ↓
/feature-implementation
```

### 决策速查表

| 项目特征 | 需要 /framework-design | 需要 /interface-design |
|---------|------------------------|------------------------|
| 新项目 + 复杂架构 | ✅ 强烈推荐 | ✅ 推荐 |
| 新项目 + 简单工具 | ❌ 可跳过 | ❌ 可跳过 |
| 现有项目 + 新增 API | ❌ 不需要 | ✅ 推荐 |
| 现有项目 + 小功能 | ❌ 不需要 | ❌ 可跳过 |
| 架构重构 | ✅ 必须 | ✅ 推荐 |
| Bug 修复 | ❌ 不需要 | ❌ 不需要 |
| 性能优化 | ⚠️ 视情况 | ❌ 不需要 |
