# LocalVideoServer 架构设计方案

## 1. 设计目标与约束

### 1.1 核心目标

为个人用户提供一个**轻量级、零依赖、单文件运行**的本地视频服务器，通过网页跨设备（手机/平板/电脑）观看和管理本地视频文件。

### 1.2 硬性约束

| 约束项 | 要求 | 理由 |
|--------|------|------|
| 编程语言 | C99 (严格标准) | 极低资源占用，无运行时依赖 |
| 数据库 | SQLite3 | 嵌入式，零配置，事务支持 |
| HTTP 服务器 | 自实现 (socket) | 零第三方库，完全控制请求流程 |
| 视频格式 | MP4, WebM | 浏览器原生支持，无需转码 |
| 用户认证 | 不需要 | 仅局域网使用 |
| 内存占用 | < 50MB | 轻量级设计目标 |
| 响应时间 | < 500ms | 页面交互流畅 |

### 1.3 设计原则

- **零外部依赖**：除系统库（SQLite3、pthread）外不引入任何第三方库
- **模块化架构**：高内聚低耦合，通过 GCC section 属性自动注册模块生命周期
- **内核风格编程**：侵入式链表、通知链等内核技术模式
- **流式传输**：大文件分块读取，不加载整个文件到内存

## 2. 架构风格

### 2.1 选择：分层单体架构 + 事件驱动通讯

**选择理由**：

1. **单体而非微服务**：项目面向个人用户，单机部署，无需分布式能力。微服务的复杂度远超需求。
2. **分层架构**：清晰的职责分离（表现层→服务层→业务层→数据层），便于理解和维护。
3. **事件驱动补充**：模块间通过通知链解耦通讯，避免紧耦合的直接函数调用。

## 3. 系统架构详细说明

### 3.1 分层职责

#### 表现层 (Web Frontend)
- **技术栈**：原生 HTML5 + CSS3 + JavaScript（无框架）
- **入口**：`web/static/index.html`
- **核心组件**：
  - 视频列表展示（网格/列表视图）
  - HTML5 Video 播放器（支持 Range Requests）
  - 搜索框（防抖输入）
  - 分类侧边栏
  - 历史记录面板
  - 收藏管理面板
  - 黑名单管理界面
  - Toast 通知系统
  - 键盘快捷键绑定
- **状态管理**：localStorage 存储用户偏好

#### 服务层 (HTTP Server)
- **入口**：[http_server.c](src/modules/http_server/http_server.c)
- **子模块**：
  - [http_request.c](src/modules/http_server/http_request.c) — HTTP 请求解析
  - [http_response.c](src/modules/http_server/http_response.c) — HTTP 响应构建
- **核心能力**：
  - Socket 监听与连接接受
  - HTTP/1.1 协议解析（GET/POST/DELETE）
  - 静态文件服务（HTML/CSS/JS）
  - 视频流式传输（分块读取）
  - Range Requests 支持（206 Partial Content）
  - CORS 头设置
  - 每连接一线程并发模型

#### 业务层 (Business Modules)

| 模块 | 路径 | 职责 |
|------|------|------|
| api_handler | `src/modules/api_handler/` | REST API 路由分发、JSON 响应生成、业务逻辑编排 |
| video_scanner | `src/modules/video_scanner/` | 目录递归扫描、视频文件识别、inotify 实时监控 |
| db_manager | `src/modules/db_manager/` | SQLite3 封装、CRUD 操作、预编译语句 |
| config | `src/modules/config/` | 命令行参数解析、配置存储 |

#### 基础设施层 (Shared Infrastructure)

| 模块 | 路径 | 职责 |
|------|------|------|
| module | `src/shared/module/` | GCC section 属性的模块生命周期管理（init/sub/run/exit） |
| notifier | `src/shared/notifier/` | 内核风格通知链（优先级注册、事件回调） |
| list | `src/shared/list/` | 侵入式双向链表（container_of 宏） |
| log | `src/shared/log/` | 多级别日志系统（DEBUG/INFO/WARNING/ERROR） |
| thread | `src/shared/thread/` | pthread 封装（互斥锁、条件变量） |
| json | `src/shared/json/` | 流式 JSON 生成器 |

### 3.2 模块化生命周期机制

```
main()
  │
  ├── config_parse_args()          ← 命令行参数解析
  │
  ├── module_init_all()            ← 初始化阶段
  │   ├── config_init()            ← 加载配置
  │   ├── db_manager_init()        ← 打开数据库连接
  │   ├── http_server_init()       ← 绑定端口
  │   └── ...                      ← 其他模块初始化
  │
  ├── module_sub_all()             ← 订阅阶段
  │   ├── api_handler_sub()        ← 注册事件监听
  │   └── ...                      ← 其他模块订阅事件
  │
  ├── module_run_all()             ← 运行阶段
  │   ├── http_server_run()        ← 启动 socket 监听循环
  │   ├── video_scanner_run()      ← 启动 inotify 监控
  │   └── ...                      ← 其他模块启动
  │
  ├── while(running) { sleep(1); } ← 主循环等待信号
  │
  └── module_exit_all()            ← 退出阶段
      ├── http_server_stop()       ← 停止监听
      ├── video_scanner_stop()     ← 停止监控
      ├── db_manager_close()       ← 关闭数据库
      └── ...                      ← 其他模块清理
```

**实现原理**：
```c
// 通过 GCC section 属性将函数指针放入自定义 ELF 段
#define MODULE_INIT(fn, name) \
    __attribute__((used, section(".embedi_init"))) \
    static const module_init_entry_t _lv_init_##fn = { name, fn }

// 链接器脚本 linker.ld 自动生成 __start_xxx / __stop_xxx 符号
// module.c 在运行时遍历符号表调用所有注册函数
```

### 5.2 表结构 DDL

```sql
-- 视频元数据表
CREATE TABLE IF NOT EXISTS videos (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    path TEXT NOT NULL UNIQUE,
    title TEXT NOT NULL,
    category TEXT NOT NULL DEFAULT '',
    size INTEGER NOT NULL DEFAULT 0,
    created_at INTEGER NOT NULL DEFAULT (strftime('%s','now')),
    blacklisted INTEGER NOT NULL DEFAULT 0
);

-- 播放历史记录表
CREATE TABLE IF NOT EXISTS history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    video_id INTEGER NOT NULL,
    video_title TEXT NOT NULL,
    video_path TEXT NOT NULL,
    position INTEGER NOT NULL DEFAULT 0,
    played_at INTEGER NOT NULL DEFAULT (strftime('%s','now')),
    FOREIGN KEY (video_id) REFERENCES videos(id) ON DELETE CASCADE
);

-- 收藏表
CREATE TABLE IF NOT EXISTS favorites (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    video_id INTEGER NOT NULL UNIQUE,
    video_title TEXT NOT NULL,
    video_path TEXT NOT NULL,
    created_at INTEGER NOT NULL DEFAULT (strftime('%s','now')),
    FOREIGN KEY (video_id) REFERENCES videos(id) ON DELETE CASCADE
);

-- 目录黑名单表
CREATE TABLE IF NOT EXISTS blacklist (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    path TEXT NOT NULL UNIQUE,
    created_at INTEGER NOT NULL DEFAULT (strftime('%s','now'))
);

-- 索引优化
CREATE INDEX IF NOT EXISTS idx_videos_category ON videos(category);
CREATE INDEX IF NOT EXISTS idx_videos_blacklisted ON videos(blacklisted);
CREATE INDEX IF NOT EXISTS idx_history_video_id ON history(video_id);
CREATE INDEX IF NOT EXISTS idx_favorites_video_id ON favorites(video_id);
```

```
    │ [video_scanner] readdir() + stat()
    ▼
SQLite 数据库 (videos 表)
    │
    │ [db_manager] SELECT + 回调遍历
    ▼
[api_handler] 构建 JSON 数组
    │
    │ [json_writer] 流式序列化
    ▼
[http_server] HTTP 200 + JSON body
    │
    ▼
浏览器前端 (fetch → JSON.parse → 渲染列表)
```

#### 核心数据流：视频播放

```
用户点击视频
    │
    ▼
前端 <video src="/video/relative/path.mp4">
    │
    │ HTTP GET /video/relative/path.mp4
    ▼
[http_server] 路径安全校验（防路径遍历）
    │
    │ open() + read() 分块（如 64KB/chunk）
    ▼
[http_response] 200 OK (或 206 Partial Content with Range)
    │ Content-Type: video/mp4
    │ Accept-Ranges: bytes
    ▼
浏览器 Video 解码播放
    │
    │ [timeupdate 事件] 每 5 秒
    ▼
前端 POST /api/history { video_id, position }
    │
    ▼
[api_handler] → [db_manager] INSERT/UPDATE history
```

#### 核心数据流：文件实时监控

```
inotify (IN_CREATE | IN_DELETE | IN_MODIFY | IN_MOVED_FROM | IN_MOVED_TO)
    │
    ▼
[video_scanner watcher thread]
    │
    ├─ 新文件 → db_manager_video_insert()
    ├─ 删除文件 → db_manager_video_delete_by_path()
    ├─ 移动文件 → delete + insert
    └─ 事件 debounce（聚合短时间内的批量事件）
    │
    ▼
[notifier_chain_notify(EVENT_VIDEO_CHANGED)]
    │
    ▼
[api_handler] 缓存失效（下次查询重新从 DB 读取）
```

### 5.4 缓存策略

| 数据类型 | 缓存策略 | 理由 |
|----------|----------|------|
| 视频列表 | 无缓存（每次查 DB） | 数据量小，查询简单 |
| 静态文件 | OS 页缓存 | 让操作系统处理 |
| 配置信息 | 全局只读变量 | 启动后不变更 |
| 搜索结果 | 无缓存 | 搜索词多变 |

**未来可选优化**：视频列表内存缓存 + inotify 失效机制

## 6. 技术组件

### 6.1 组件清单

| 组件 | 版本要求 | 选型理由 |
|------|----------|----------|
| GCC | ≥ 6.0 | section 属性支持 |
| SQLite3 | ≥ 3.0 | 嵌入式数据库，事务支持 |
| pthread | POSIX | 线程并发 |
| CMake | ≥ 3.10 | 跨平台构建 |
| inotify | Linux 2.6.13 | 文件系统实时监控 |

### 6.2 关键技术决策

#### 决策 1：GCC Section 属性实现模块自注册

**问题**：如何避免手动维护模块初始化列表？
**方案**：利用 `__attribute__((section()))` 将函数指针放入自定义 ELF 段，链接器自动生成边界符号。

**优势**：
- 新增模块只需在 .c 文件中添加 `MODULE_xxx()` 宏，无需修改其他文件
- 编译器/链接器保证不会遗漏
- 支持独立编译和链接

**约束**：仅适用于 GCC/Clang，不兼容 MSVC（本项目不关心）

#### 决策 2：每连接一线程模型

**问题**：如何处理 HTTP 并发请求？

**方案**：主线程 accept()，每个连接 fork 一个 pthread 处理。

**优势**：
- 实现简单，每个请求有独立栈空间
- 逻辑清晰，无需状态机
- 适合局域网场景（并发量 < 100）

**权衡**：高并发下线程开销较大，但本项目的使用场景不需要 epoll 异步模型

#### 决策 3：通知链 vs 直接函数调用

**问题**：模块间如何通讯？

**方案**：混合模式——常规操作用直接函数调用，事件通知用通知链。

| 场景 | 方式 | 示例 |
|------|------|------|
| API 请求数据 | 直接调用 | api_handler → db_manager_video_get_all() |
| 扫描完成通知 | 通知链 | video_scanner → NOTIFY_EVENT_SCANNED |
| 播放事件 | 通知链 | api_handler → NOTIFY_PLAYBACK_STARTED |

## 7. 关键质量属性

### 7.1 性能保证

| 指标 | 目标值 | 保证措施 |
|------|--------|----------|
| 页面响应时间 | < 500ms | SQLite 索引优化，静态文件直接 sendfile |
| 视频拖动延迟 | < 200ms | Range Requests，seek 后立即返回对应分块 |
| 内存占用 | < 50MB | 流式传输，不缓冲大文件；视频列表按需加载 |
| 启动时间 | < 2s | 延迟扫描（先启动服务，后台扫描） |

### 7.2 可靠性措施

| 风险 | 应对策略 |
|------|----------|
| 客户端断开连接 | SIGPIPE 忽略 + send()/write() 返回值检查 |
| 数据库锁定冲突 | SQLite WAL 模式 + busy_timeout |
| 文件被删除/移动 | inotify 监控 + 数据库同步更新 |
| 信号中断 | SIGINT/SIGTERM 优雅关闭（module_exit_all） |
| 大目录扫描超时 | 分批提交数据库，避免长事务 |

### 7.3 安全策略

| 威胁 | 防护措施 |
|------|----------|
| 路径遍历攻击 | 视频路径规范化校验，禁止 `..` 跳出根目录 |
| SQL 注入 | 全部使用 SQLite3 预处理语句（sqlite3_bind_*） |
| 缓冲区溢出 | 使用 snprintf 替代 sprintf，固定大小结构体字段 |
| 信息泄露 | HTTP 错误响应不暴露内部细节 |
| CORS | 设置 `Access-Control-Allow-Origin: *`（局域网可接受） |

### 7.4 可扩展性设计

| 维度 | 当前方案 | 扩展路径 |
|------|----------|----------|
| 新增功能模块 | 添加 `MODULE_xxx()` 宏即可 | 零侵入扩展 |
| 新增 API 端点 | 在 api_handler 添加路由规则 | 路由表驱动，易扩展 |
| 新增数据表 | db_manager 添加 CRUD 函数 | 回调模式统一接口 |
| 前端新页面 | web/static/ 下添加 HTML | 静态文件服务器自动覆盖 |
| 视频格式扩展 | video_scanner 添加新的扩展名判断 | 格式检测函数可插拔 |

## 8. 线程模型

### 8.1 线程拓扑

```
主线程 (main)
  ├── 信号处理 (SIGINT/SIGTERM)
  ├── 模块生命周期管理 (init/sub/run/exit)
  └── 主循环 sleep(1)

  ┌── HTTP 服务线程 (http_server_run)
  │     ├── socket bind + listen
  │     └── accept() 循环
  │           └── [每次 accept] → 创建工作线程
  │                 └── 工作线程 (per-connection)
  │                       ├── 解析 HTTP 请求
  │                       ├── 路由到 api_handler 或静态文件
  │                       ├── 发送响应
  │                       └── 退出
  │
  └── 文件监控线程 (video_scanner start_watcher)
        ├── inotify_init()
        └── epoll_wait() 循环
              ├── IN_CREATE → 插入数据库
              ├── IN_DELETE → 从数据库删除
              └── IN_MODIFY → 更新元数据
```

### 8.2 线程安全机制

| 共享资源 | 保护方式 | 说明 |
|----------|----------|------|
| SQLite3 连接 | 互斥锁 | SQLite3 线程安全但串行执行 |
| 全局配置 | 只读（初始化后不变） | 无需保护 |
| 日志系统 | 互斥锁或线程局部存储 | 防止输出交错 |
| 通知链 | 互斥锁 | 注册/注销/通知操作 |

```c
typedef enum {
    /* 视频相关事件 */

    /* 播放相关事件 */
    EVENT_PLAYBACK_STARTED,        /* 开始播放 */
    EVENT_PLAYBACK_STOPPED,        /* 停止播放 */

    EVENT_FAVORITE_ADDED,          /* 添加收藏 */

    EVENT_BLACKLIST_ADDED,         /* 目录加入黑名单 */
    EVENT_BLACKLIST_REMOVED,       /* 目录移除黑名单 */

    EVENT_COUNT
} event_type_t;
```

## 11. 风险与应对

### 11.1 技术风险

| 风险 | 可能性 | 影响 | 应对策略 |
|------|--------|------|----------|
| GCC section 属性不可移植 | 低 | 中 | 项目明确限定 GCC/Clang，不接受 MSVC |
| SQLite3 并发写瓶颈 | 中 | 低 | 使用 WAL 模式 + busy_timeout；局域网场景并发低 |
| inotify fd 耗尽（watch 过多） | 低 | 中 | 限制 watch 深度；对大目录采用递归 watch |
| 大文件（>4GB）视频 | 低 | 中 | 使用 int64_t/off_t 处理偏移量 |
| HTTP 工作线程泄漏 | 中 | 高 | 确保 finally 块中关闭 fd 和释放资源 |

### 11.2 业务风险

| 风险 | 应对策略 |
|------|----------|
| 需求蔓延（转码/上传/多用户） | 严格遵循项目边界文档，Phase 1 不做超出范围的功能 |
| 视频文件名编码问题（中文/特殊字符） | URL 编码 + UTF-8 处理 |
| 前端兼容性（旧浏览器） | 目标现代浏览器（Chrome/Firefox/Safari 近两个版本） |
