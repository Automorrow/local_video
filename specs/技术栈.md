# 技术栈选型

## 1. 核心决策 (Key Decisions)
*简述为什么选择这套技术栈？*
基于项目对**低依赖、轻量级、高性能**的严格要求，选择C语言作为后端开发语言，配合SQLite3嵌入式数据库和自实现HTTP服务器。采用模块化初始化机制（GCC section属性 + module_init宏）提升代码可维护性和扩展性。

## 2. 前端 (Frontend)
*   **框架**: 原生 HTML5 + CSS3 + JavaScript（无框架依赖）
*   **样式方案**: 原生 CSS（Flexbox/Grid 布局）
*   **状态管理**: 原生 JavaScript（localStorage 存储本地偏好）
*   **视频播放**: HTML5 Video API（原生支持 MP4/WebM）

## 3. 后端 (Backend)
*   **运行时/语言**: C语言 (C99标准)
*   **Web 框架**: 自实现 HTTP/1.1 服务器（基于 socket）
*   **API 协议**: RESTful API（JSON 格式）
*   **并发模型**: POSIX 线程 (pthread) - 每个连接一个线程
*   **模块化机制**: GCC section 属性 + 完整生命周期宏
    - `module_init(fn)`: 模块初始化函数（启动时调用）
    - `module_sub(fn)`: 模块订阅函数（初始化完成后注册通知链）
    - `module_run(fn)`: 模块运行函数（主循环阶段调用）
    - `module_exit(fn)`: 模块退出函数（清理资源）
    - 自定义段 `.embedi_init` / `.embedi_sub` / `.embedi_run` / `.embedi_exit` 分类存储函数指针
    - 链接器符号遍历自动按顺序调用各阶段函数
    - 支持优先级控制（可选）

## 4. 数据存储 (Database)
*   **主数据库**: SQLite3（嵌入式，零配置）
    - 存储视频元数据、目录结构
    - 存储播放历史记录
    - 存储收藏列表

## 5. 基础设施 & 工具 (Infra & Tools)
*   **构建系统**: CMake
*   **编译器**: GCC / Clang
*   **版本控制**: Git
*   **配置方式**: 命令行参数 + 可选配置文件

## 6. 关键技术实现细节

### 6.1 HTTP 服务器特性
- 支持 HTTP/1.1 GET 请求
- 支持 Range Requests（视频拖动播放）
- 支持静态文件服务（HTML/CSS/JS）
- 支持视频流式传输

### 6.2 模块划分
- `http_server`: HTTP 服务模块
- `video_scanner`: 视频扫描与元数据提取
- `db_manager`: SQLite3 数据库管理
- `api_handler`: REST API 处理
- `config`: 配置管理

### 6.3 模块化生命周期机制

完整的模块生命周期管理：

1. **初始化阶段 (Init)**
   - 调用所有 `module_init()` 注册的函数
   - 用途：配置加载、数据库连接、资源预分配

2. **订阅阶段 (Sub)**
   - 调用所有 `module_sub()` 注册的函数
   - 用途：注册通知链事件监听器

3. **运行阶段 (Run)**
   - 调用所有 `module_run()` 注册的函数
   - 用途：启动HTTP服务、进入主事件循环

4. **退出阶段 (Exit)**
   - 调用所有 `module_exit()` 注册的函数
   - 用途：关闭数据库、释放资源、清理连接

实现原理：
- 利用 GCC `__attribute__((section(".embedi_xxx")))` 将函数指针放入自定义段
- 利用 GCC `__attribute__((used))` 防止编译器优化掉未引用的全局变量
- 链接器自动生成段起始和结束符号（`__start_xxx` / `__stop_xxx`）
- 运行时遍历符号表自动调用所有注册函数

### 6.4 模块间通讯机制

采用**通知链 (Notifier Chain)** 机制实现模块间解耦通讯：

- **通知链结构**：基于内核风格侵入式链表实现
- **事件类型**：定义统一的事件枚举（如 `VIDEO_SCANNED`、`PLAYBACK_STARTED` 等）
- **注册机制**：模块通过 `notifier_register()` 注册感兴趣的事件
- **触发机制**：事件产生方通过 `notifier_call()` 触发通知
- **优先级支持**：通知处理函数支持优先级，按顺序执行
- **中断支持**：处理函数可返回状态中断通知链传递

应用场景：
- 视频扫描完成后通知API模块更新缓存
- 播放开始时通知历史记录模块记录
- 收藏变化时通知数据库模块更新

### 6.5 任务模型：串行与并行

#### 6.5.1 串行任务（主线程执行）
- **执行位置**：`module_run()` 阶段，主事件循环中
- **适用场景**：
  - HTTP请求的同步处理（轻量级）
  - 状态机转换
  - 配置重新加载
  - 通知链回调执行

#### 6.5.2 并行任务（独立线程执行）
- **执行方式**：模块内部创建独立 pthread 线程
- **线程池**：可选择使用线程池管理（可选）
- **适用场景**：
  - 视频文件扫描（耗时IO操作）
  - HTTP服务主循环（持续监听）
  - 视频元数据提取
  - 后台数据库清理

#### 6.5.3 线程安全机制
- **互斥锁 (pthread_mutex_t)**：保护共享数据结构
- **条件变量 (pthread_cond_t)**：线程间同步
- **原子操作**：轻量级计数器等场景
- **无锁数据结构**：只读数据使用（如配置信息）
