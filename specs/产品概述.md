# 产品概述

## 1. 项目愿景 (Vision)
*用一句话概括：我们要为谁，提供什么核心价值？*
> 为个人用户提供一个轻量级的本地视频服务器，实现在不同设备（手机/平板/电脑）上通过网页便捷地观看和管理本地视频文件。

## 2. 核心价值 (Value Proposition)
*   **用户价值**：无需复杂的媒体服务器软件，即可在家庭局域网内跨设备共享本地视频，支持搜索、分类、历史记录和收藏，提升视频观看体验。
*   **商业/技术价值**：采用C语言 + SQLite3的技术栈，实现极低的系统资源占用和依赖，无需安装任何运行时环境，单文件即可运行。

## 3. 核心板块 (Core Modules)
*将系统拆解为关键的顶层模块（建议 2-4 个核心板块）。*

### 3.1 HTTP服务模块
*   **描述**：提供Web界面访问和视频文件流式传输
*   **关键特性**：
    *   自定义HTTP服务器实现，无第三方库依赖
    *   支持视频文件范围请求（Range Requests）实现拖动播放
    *   静态资源（HTML/CSS/JS）服务
    *   支持 DELETE 方法用于 REST API

### 3.2 视频管理模块
*   **描述**：视频文件扫描、元数据管理和检索
*   **关键特性**：
    *   扫描固定目录及其子目录的视频文件
    *   按目录自动分类
    *   视频搜索功能
    *   SQLite3存储视频元数据
    *   inotify + epoll 实时文件监控
    *   数据库为空时自动扫描
    *   目录黑名单管理（标记而非删除，可快速恢复）

### 3.3 播放功能模块
*   **描述**：视频播放、历史记录和收藏管理
*   **关键特性**：
    *   浏览器原生格式播放（MP4, WebM）
    *   播放历史记录（支持继续上次播放位置）
    *   收藏功能（带收藏标记）
    *   随机播放（自动播放、按钮、自动下一个）
    *   Toast 通知提示
    *   加载动画
    *   键盘快捷键支持

## 4. 目标用户 & 核心场景 (User & Scenes)
### 4.1 目标用户
*   **个人用户**：拥有本地视频文件，需要在手机、平板、电脑等多设备上观看的技术爱好者

### 4.2 核心场景 (Key Use Cases)
*   **场景 1**：个人用户 在电脑上启动服务器后，通过手机浏览器访问，随机播放一个视频文件。
*   **场景 2**：个人用户 在平板上通过搜索功能找到想看的视频，播放后自动记录到历史记录中，并添加到收藏。
*   **场景 3**：个人用户 在观看完一个视频后，系统自动随机播放下一个视频。
*   **场景 4**：个人用户 从历史记录播放视频时，系统提示是否继续上次播放位置。
*   **场景 5**：个人用户 将某个目录加入黑名单，该目录下的视频不再显示，删除黑名单后可快速恢复。

## 5. 关键业务流程 (Core Workflows)
*描述系统最核心的 1-2 条业务链路，这决定了技术架构的复杂度。*
> 1. 服务器启动 → 检查数据库是否为空 → 为空则扫描 → 实时监控文件变化 → 启动HTTP服务
> 2. 用户访问首页 → 选择视频 → 显示继续播放对话框 → 流式传输视频 → 记录播放历史
> 3. 用户管理黑名单 → 添加/删除目录 → 标记/取消标记视频 → 自动刷新视频列表

## 6. 验收标准 (Acceptance Criteria)
*项目第一阶段结束的具体标志是什么？（用于防止需求蔓延）*
*   **功能完备性**：视频列表展示、搜索、分类、播放、历史记录、收藏、随机播放、黑名单管理功能全部可用
*   **技术指标**：支持视频拖动播放，页面响应 < 500ms，单进程内存占用 < 50MB
*   **用户体验**：Toast 通知、防抖搜索、继续播放对话框、收藏标记、加载动画、键盘快捷键

## 7. 约束条件 (Constraints)
*非功能性的硬性限制。*
*   后端必须使用C语言实现
*   数据库必须使用SQLite3
*   HTTP服务器必须自己实现，不使用第三方Web框架
*   仅支持浏览器原生可播放的视频格式（MP4, WebM）
*   不需要用户认证

## 8. 项目边界 (Out of Scope)
*明确本阶段**不做**什么，避免需求蔓延。*
*   暂不支持视频转码
*   暂不开发移动端App
*   暂不支持多用户管理
*   暂不支持视频上传功能
