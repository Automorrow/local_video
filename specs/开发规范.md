## 4. C 语言特定规范

### 4.1 头文件包含守卫

```c
// ✅ 正确写法
#ifndef HTTP_SERVER_H
#define HTTP_SERVER_H

/* 头文件内容 */

#endif /* HTTP_SERVER_H */

// ❌ 错误写法
#pragma once
```

### 4.2 静态函数 vs 全局函数

| 场景 | 偏好 | 原因 |
|------|------|------|
| 模块内部使用 | `static` | 减少符号可见性，避免命名冲突 |
| 对外暴露 | 无 static | 明确的公共 API |

### 4.3 避免的写法

```c
// ❌ 避免：使用不安全的字符串函数
char buffer[100];
strcpy(buffer, input);
sprintf(buffer, "Value: %d", value);

// ✅ 替代方案：使用安全函数
char buffer[100];
strncpy(buffer, input, sizeof(buffer) - 1);
buffer[sizeof(buffer) - 1] = '\0';
snprintf(buffer, sizeof(buffer), "Value: %d", value);
```

```c
// ❌ 避免：不检查 malloc 返回值
char *ptr = malloc(size);
strcpy(ptr, data);

// ✅ 替代方案：检查返回值
char *ptr = malloc(size);
if (ptr == NULL) {
    /* 错误处理 */
    return ERROR;
}
strcpy(ptr, data);
```

```c
// ❌ 避免：全局变量过度使用
int g_state;
void func() { g_state = 1; }

// ✅ 替代方案：封装在结构体中
typedef struct {
    int state;
} Context;
void func(Context *ctx) { ctx->state = 1; }
```

```c
// ❌ 避免：魔法数字
if (status == 42) { ... }

// ✅ 替代方案：使用枚举或宏定义
#define STATUS_OK 0
#define STATUS_ERROR 1
if (status == STATUS_OK) { ... }
```

### 4.4 内存管理规范

- 动态内存分配后必须检查返回值
- 使用完后必须释放，避免内存泄漏
- 避免频繁的 malloc/free，考虑使用内存池（可选）

## 5. 模块化编程范式规范（内核风格编程技术

### 5.1 函数指针结构体（面向对象实现）

```c
// 正确示例
typedef struct {
    int (*init)(void *ctx);
    int (*start)(void *ctx);
    void (*exit)(void *ctx);
} ModuleOps;

typedef struct {
    ModuleOps ops;
    void *data;
} Module;
```

### 5.2 侵入式链表

参考内核风格侵入式链表，节点直接嵌入结构体：

```c
typedef struct list_node {
    struct list_node *next;
    struct list_node *prev;
} list_node_t;

typedef struct {
    char *name;
    list_node_t node;  /* 嵌入链表节点 */
} Video;
```

### 5.3 通知链机制

使用通知链实现模块间事件通讯：

```c
typedef struct notifier_node {
    int (*callback)(void *data);
    int priority;
    list_node_t node;
} notifier_node_t;
```

### 5.4 模块自注册机制

使用 GCC section 属性实现模块自注册：

```c
typedef void (*init_t)(void);

#define module_init(fn) \
    static init_t __init_##fn __attribute__((used, section(".embedi_init"))) = fn
```

### 5.5 模块边界与文件拆分规则

- 对外能力只通过模块头文件暴露；模块内部拆分出的实现文件不得新增跨模块公共入口。
- 允许在同一模块目录下按职责拆分多个 `.c` 文件，但这些文件必须通过 `*_internal.h` 共享内部声明，不能把内部辅助函数泄漏到 `src/include/`。
- 一个实现文件只承载一类职责：例如 `http_server` 可拆为连接循环、静态文件、缩略图、视频流；禁止继续在单文件内混放网络监听、路由分发、媒体发送三类逻辑。
- 所有非公共符号默认 `static`；若必须被同模块多个实现文件复用，应先评估是否应沉淀到 `src/shared/`。
- 收敛优先于重写：优先抽取重复逻辑、统一已有工具函数、缩小文件职责，不以“推翻重做”替代渐进整理。

### 5.6 生命周期规则

- `MODULE_INIT` 仅做轻量初始化：读取配置、准备状态、打开句柄，不启动长期阻塞循环。
- `MODULE_SUB` 仅做事件/通知订阅；若模块无订阅行为，可以省略该阶段注册。
- `MODULE_RUN` 才能启动线程、socket 监听、文件监控等持续运行行为。
- `MODULE_EXIT` 必须幂等，允许在部分初始化失败或重复退出场景下安全调用。
- 模块生命周期顺序固定为 `init -> sub -> run -> exit`，新增模块时必须在文档和代码中同时保持该约束。

### 5.7 仓库边界规则

- Git 仓库只保留源代码、文档、测试和构建脚本；运行产物、数据库、依赖下载目录、临时日志不得作为项目内容提交。
- 本地构建输出统一放在 `build/`；前端依赖目录若仅用于本地工具链，应通过 `.gitignore` 隔离。
- 新增目录前必须先判断它是“源码输入”还是“运行输出”；运行输出不得放入 `src/`、`specs/`、`tests/`、`web/static/`。

### 5.8 模块内分层收敛规则

- 当模块同时承担“入口路由/生命周期”“资源端点逻辑”“通用响应或解析辅助”三类职责时，应优先在模块目录内拆分，而不是继续堆积到单个 `.c` 文件中。
- 模块主文件应优先保留对外入口、路由分发和生命周期注册；资源级处理逻辑应下沉到独立实现文件；可复用的响应拼装、请求解析、缓冲区辅助等代码应收敛到模块内部共享实现。
- `*_internal.h` 只声明模块内部复用的类型和函数，不得替代公共头文件对外暴露能力。
- 这类拆分以“行为不变、公共接口不变、依赖方向更清晰”为验收标准，不以文件数量增加为目标。

## 7. Git 工作流

### 7.1 分支策略

> 独立开发者，采用简化 Trunk-Based

```
main (始终可部署)
  └── feature/xxx (功能分支，从 main 分出，合并回 main)
```

| 规则 | 说明 |
|------|------|
| **主分支** | `main`，始终可编译运行 |
| **功能分支** | 从 `main` 分出，功能完成后合并回 `main` |
| **分支生命周期** | 1-3 天 |
| **合并方式** | 直接合并或 Squash Merge |

### 7.2 分支命名规范

```
<type>/<short-description>
```

| 类型 | 用途 | 示例 |
|------|------|------|
| `feat/` | 新功能 | `feat/video-random-play` |
| `fix/` | Bug 修复 | `fix/http-range-request` |
| `refactor/` | 重构 | `refactor/module-init` |
| `docs/` | 文档更新 | `docs/api-spec` |
| `chore/` | 构建/配置 | `chore/cmake-config` |

### 7.3 提交信息规范 (Conventional Commits)

所有 Git 提交信息**必须**遵循 [Conventional Commits](https://www.conventionalcommits.org/) 格式：

```
<type>(<scope>): <subject>

[body]

[footer]
```

#### 类型 (type)

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | |
| `fix` | Bug 修复 | |
| `refactor` | 重构（不改变功能） | |
| `style` | 代码格式调整 | |
| `docs` | 文档变更 | |
| `test` | 测试相关 | |
| `chore` | 构建/工具/依赖 | |
| `perf` | 性能优化 | |

#### 作用域 (scope)

| scope | 对应范围 |
|-------|---------|
| `http` | HTTP 服务模块 |
| `video` | 视频扫描模块 |
| `db` | 数据库管理模块 |
| `api` | API 处理模块 |
| `config` | 配置管理模块 |
| `shared` | 共享基础设施 |

#### 提交信息规则

- `subject`：中文、小写开头、不用句号、50 字符以内
- `body`：解释**为什么**，不是**做了什么**
- `footer`：Issue 引用格式（如 `Closes #123`）

#### 示例

```
feat(http): 支持视频范围请求

实现 HTTP/1.1 Range Requests 支持，允许用户拖动视频进度条播放。
使用分段读取文件的方式，避免加载整个大文件。

Closes #42
```

```
fix(db): 修复收藏记录重复添加的问题

根因：检查收藏是否存在时，SQL 查询条件写错了。
修复：修正 SQL 查询，使用正确的列名进行匹配。
```

## 8. 代码注释规范

### 8.1 何时写注释

| 需要注释 | 不需要注释 |
|----------|-----------|
| 解释"为什么"这么做 | 解释"做了什么"（代码本身能说明的 |
| 复杂的算法或逻辑 | 简单明显的代码 |
| 非明显的边界条件处理 | 标准的 API 调用 |
| TODO/FIXME 标记 | 复述代码的注释 |

### 8.2 注释风格

```c
// ✅ 解释 "为什么"
/* 必须检查 malloc 返回值，否则可能导致段错误
if (ptr == NULL) {
    return -1;
}

// ❌ 描述 "做了什么"
/* 检查 ptr 是否为空
if (ptr == NULL) {
    return -1;
}
```

### 8.3 函数注释

```c
/**
 * @brief 启动 HTTP 服务器
 * @param port 监听端口号
 * @param web_root 静态文件根目录
 * @return 成功返回 0，失败返回错误码
 */
int http_server_start(int port, const char *web_root);
```

### 8.4 TODO / FIXME / HACK 标记

```c
// TODO: 实现线程池优化，减少线程创建开销
// FIXME: 内存泄漏，需要在退出时释放 resource
// HACK: 临时解决方案，生产环境需重构
```

## 9. 错误处理规范

### 9.1 错误码定义

使用枚举统一管理错误码：

```c
typedef enum {
    LV_OK = 0,
    LV_ERROR_INVALID_ARG,
    LV_ERROR_MEMORY,
    LV_ERROR_IO,
    LV_ERROR_DB,
} lv_error_t;
```

### 9.2 错误处理原则

- 函数返回错误码，不使用全局变量
- 调用函数后必须检查返回值
- 错误处理应及时，避免错误传播
- 关键操作使用断言检查

### 9.3 日志系统

使用共享模块记录错误和日志系统：

```c
log_error("Failed to open database: %s", error_msg);
log_info("HTTP server started on port %d", port);
```

### 9.4 安全原则

- 永远不在 HTTP 响应中暴露内部错误信息
- 所有外部输入必须验证
- 避免缓冲区溢出

## 15. 安全编码规范

| 类别 | 规则 |
|------|------|
| **输入校验** | 所有外部输入（HTTP 请求、文件路径）必须验证 |
| **缓冲区溢出防护** | 使用 `strncpy`、`snprintf` 等安全函数 |
| **SQL 注入防护** | 使用 SQLite3 预处理语句（`sqlite3_prepare_v2`） |
| **路径遍历防护** | 验证视频文件路径，防止访问配置目录外的文件 |
| **内存安全** | 动态内存分配后检查返回值，使用后及时释放 |
| **线程安全** | 共享数据必须加锁保护 |
