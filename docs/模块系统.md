# 模块系统说明

LocalVideoServer 采用“模块自注册 + 生命周期调度”的组织方式，以减少集中式注册表维护成本。

## 1. 生命周期阶段

模块按固定顺序参与四个阶段：

1. `init`：轻量初始化，准备状态或句柄
2. `sub`：注册通知链或事件订阅
3. `run`：启动线程、监听、监控等持续运行行为
4. `exit`：安全退出，必须幂等

对应调度入口在 `src/shared/module/module.c` 中：

- `module_init_all()`
- `module_sub_all()`
- `module_run_all()`
- `module_exit_all()`

## 2. 模块如何自注册

模块通过 `src/shared/module/module.h` 中的宏进行注册：

- `MODULE_INIT(fn, module_name)`
- `MODULE_SUB(fn, module_name)`
- `MODULE_RUN(fn, module_name)`
- `MODULE_EXIT(fn, module_name)`

这些宏会把函数指针和模块名放进特定链接段：

- `.embedi_init`
- `.embedi_sub`
- `.embedi_run`
- `.embedi_exit`

因此新增模块时，不需要再改一个全局注册表；只要在模块实现文件里声明对应 `MODULE_*` 宏即可。

## 3. `linker.ld` 的作用

Linux 构建会通过 `linker.ld` 定义上述段的起止符号：

- `__start_embedi_init` / `__stop_embedi_init`
- `__start_embedi_sub` / `__stop_embedi_sub`
- `__start_embedi_run` / `__stop_embedi_run`
- `__start_embedi_exit` / `__stop_embedi_exit`

`module.c` 通过遍历这些区间，依次调用已注册的函数。

## 4. 平台差异

- Linux 构建会在 `CMakeLists.txt` 中为主程序和测试目标添加 `-T ${CMAKE_SOURCE_DIR}/linker.ld`
- Windows 构建不走同样的链接段机制，因此当前逻辑通过条件编译与构建分支做兼容

这也是为什么：

- `linker.ld` 属于主产品构建链的一部分
- 模块生命周期机制需要在 `README.md` / `specs/项目结构.md` / 本文档中保持一致说明

## 5. 新增模块时的约束

新增模块时请同时满足以下规则：

1. 对外能力只经模块头文件暴露
2. 模块内部多职责拆分优先放在同模块目录内，通过 `*_internal.h` 共享内部声明
3. `MODULE_INIT` 不启动长期阻塞行为
4. `MODULE_RUN` 才启动线程、socket 监听、文件监控等持续任务
5. `MODULE_EXIT` 必须可重复调用且安全

若你只需要了解如何新增一个模块，请先阅读：

- `README.md`
- `specs/项目结构.md`
- `specs/开发规范.md`
- 本文档
